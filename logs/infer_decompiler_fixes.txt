1. DECOMPRESSOR LOCATION AND ENTRY POINT
========================================================================

The decompressor IS the DLL entry point (DllMain):
  RVA: 0x3C4F30
  File offset: 0x70330 (in .rsrc section, VA 0x355000 + offset 0x6FF30)
  Signature: void __fastcall DllMain(HINSTANCE hinst, DWORD reason, LPVOID reserved)

When Windows loads the DLL:
  rcx = DLL base address (0x180000000 with ASLR)
  rdx = 1 (DLL_PROCESS_ATTACH)
  r8  = 0 (reserved)

The code checks `cmp dl, 1` at 0x3C4F3F. If DLL_PROCESS_ATTACH, it enters
the decompression path. Otherwise, it returns TRUE via 0x3C51B4.

There is NO separate bootstrap stub — the entry point directly decompresses.

3. REGISTER SETUP (0x3C4F48-0x3C4F63)
========================================================================

  push rbx, rsi, rdi, rbp          ; save callee-saved regs
  lea rsi, [rip-0x6FF53]           ; rsi = 0x355000 (.rsrc1 start)
  lea rdi, [rsi-0x354000]          ; rdi = 0x6000 (output buffer in .text)
  push rdi                         ; save output ptr (popped at 0x3C5065)
  xor ebx, ebx                     ; ebx = 0 (bit accumulator)
  xor ecx, ecx                     ; ecx = 0 (prev_len)
  or rbp, -1                       ; rbp = -1 (last_offset = -1)
  call 0x3C4FB8                    ; setup r11 = getbit function

CRITICAL: rsi = 0x355000 = file offset 0x400.
  RIP after lea = 0x3C4F53
  Displacement = sign_extend(0xFFF900AD) = -458,579
  EA = 0x3C4F53 + (-458,579) = 3,493,888 = 0x355000  ✓

4. GETBIT FUNCTION (0x3C4FB8 / 0x3C4F68)
========================================================================

Two entry points share the same logic:

A) Setup entry (called once at 0x3C4F63):
   0x3C4FB8: cld
   0x3C4FB9: pop r11          ; r11 = return address (0x3C4F68)
   0x3C4FBB: jmp 0x3C4FC5     ; enter main loop

B) Fast entry (called via `call r11` during match decoding):
   0x3C4F68: add ebx, ebx     ; shift left, CF = MSB
   0x3C4F6A: je 0x3C4F6E      ; if ebx==0, reload
   0x3C4F6C: repz ret         ; return (CF = extracted bit)
   0x3C4F6E: mov ebx, [rsi]   ; reload 4 bytes from source
   0x3C4F70: sub rsi, -4      ; rsi += 4
   0x3C4F74: adc ebx, ebx     ; shift left with CF
   0x3C4F76: mov dl, [rsi]    ; preload next byte
   0x3C4F78: repz ret         ; return (CF = extracted bit)

Bit extraction: MSB-first via carry flag.
  CF=1 after shift → bit is 1 → LITERAL (in main loop)
  CF=0 after shift → bit is 0 → MATCH (in main loop)

5. MAIN LOOP (0x3C4FC5)
========================================================================

  0x3C4FC5: mov dl, [rsi]      ; preload byte at current source
  0x3C4FC7: add ebx, ebx       ; shift left, CF = MSB
  0x3C4FC9: jne 0x3C4FD5       ; if ebx!=0, skip reload
  0x3C4FCB: mov ebx, [rsi]     ; reload 4 bytes
  0x3C4FCD: sub rsi, -4        ; rsi += 4
  0x3C4FD1: adc ebx, ebx       ; shift left with CF (consumes 2nd bit)
  0x3C4FD3: mov dl, [rsi]      ; preload next byte
  0x3C4FD5: jb 0x3C4FBD        ; CF=1 → LITERAL, CF=0 → MATCH

LITERAL path (0x3C4FBD):
  0x3C4FBD: inc rsi            ; advance source
  0x3C4FC0: mov [rdi], dl      ; store byte
  0x3C4FC2: inc rdi            ; advance dest
  (falls through to 0x3C4FC5)

MATCH path (0x3C4FD7):
  0x3C4FD7: lea eax, [rcx+1]   ; eax = prev_len + 1
  0x3C4FDA: jmp 0x3C4FE3       ; enter length decode

========================================================================
6. LENGTH DECODE (0x3C4FDC-0x3C4FF6) — Gamma Coding
========================================================================

  0x3C4FE3: call r11           ; get bit → eax = 2*eax + CF
  0x3C4FE6: adc eax, eax
  0x3C4FE8: add ebx, ebx       ; get another bit (inline)
  0x3C4FEA: jne 0x3C4FF6
  ... (reload if ebx==0) ...
  0x3C4FF6: jae 0x3C4FDC       ; CF=0 → continue loop
  ; CF=1 → stop bit, exit loop

  0x3C4FF8: sub eax, 3         ; eax -= 3
  0x3C4FFB: jb 0x3C5016        ; if eax<3 → short match

LONG MATCH (eax >= 3):
  0x3C4FFD: shl eax, 8
  0x3C5000: movzx edx, dl
  0x3C5003: or eax, edx        ; eax = (eax << 8) | dl
  0x3C5005: inc rsi            ; advance source
  0x3C5008: xor eax, 0xFFFFFFFF ; eax ^= 0xFFFFFFFF
  0x3C500B: je 0x3C5065        ; if eax==0 → TERMINATION
  0x3C500D: sar eax, 1         ; arithmetic shift right
  0x3C500F: movsxd rbp, eax    ; rbp = sign-extended offset
  0x3C5012: jb 0x3C504C        ; CF (LSB of offset) → long match

SHORT MATCH (eax < 3):
  0x3C5016-0x3C504A: decode prev_len via bit reads

MATCH LENGTH ADJUSTMENT (0x3C5051-0x3C5058):
  cmp rbp, -0x500
  adc ecx, 2                    ; if offset >= -0x500: len += 2; else: len += 3

COPY (0x3C505B → call 0x3C4F7A):
  0x3C4F7A: lea rax, [rdi+rbp] ; source = dest + negative_offset
  ... (optimized dword/byte copy loop) ...

After copy:
  0x3C5060: jmp 0x3C4FC5       ; back to main loop

8. POST-PROCESSING PASSES
========================================================================

PASS 1 — Relocation fixups (0x3C5065-0x3C50B9):
  Scans output buffer from offset 0xB8200 looking for:
  - x86 CALL instructions (E8 xx xx xx xx) → fix relative addresses
  - REX-prefixed instructions in 0x80-0x8F range
  Uses 0x27 as the fixup marker byte.

PASS 2 — Import resolution (0x3C50BB-0x3C5110):
  Reads import table at rsi+0x3C1000
  Calls [rip+0x43310] (likely LdrLoadDll or GetProcAddress)
  Resolves function names via string search + [rip+0x432FA]

PASS 3 — Base relocation (0x3C511B-0x3C5155):
  Scans relocation table, applies base fixups using bswap + add

PASS 4 — Memory protection (0x3C5155-0x3C519A):
  Calls [rip+0x432A8] (likely NtProtectVirtualMemory) twice
  Sets page permissions on the decompressed .text section

10. THE DIVERGENCE PROBLEM
========================================================================

Compressed data at file offset 0x400:
  F6 6F FF FF 48 83 EC 28 E8 27 00 04 7C 48 8D 0D 40 7D 0B 00 ...

Ground truth first 20 bytes:
  48 83 EC 28 E8 77 04 00 00 48 8D 0D 40 7D 0B 00 48 83 C4 28

Decompressor output (first 26 bytes):
  48 83 EC 28 E8 27 00 04 7C 48 8D 0D 40 7D 0B 00 1E C4 28 E9 27 08 AE B4 CC 01

Byte-by-byte comparison:
  Pos  GT    Out   Match?
  0    48    48    ✓
  1    83    83    ✓
  2    EC    EC    ✓
  3    28    28    ✓
  4    E8    E8    ✓
  5    77    27    ✗ ← FIRST DIVERGENCE
  6    04    00    ✗
  7    00    04    ✗
  ...

Root cause: First dword 0xFFFF6FF6 has 16 leading 1-bits (MSB-first).
This produces 16 consecutive LITERALs from compressed[4:20].
The ground truth needs a MATCH after only 5 LITERALs.

For the 6th bit to be 0 (MATCH), the first dword would need to be:
  111110xx... (binary, MSB-first) = 0xF8000000 | xxx

But the actual first dword is 0xFFFF6FF6 = 11111111111111110110111111110110.
The 6th bit (position 26) is 1, not 0.

========================================================================

1. The decompressor at 0x3C4F30 is the DLL entry point (DllMain).
2. It reads compressed data from rsi = 0x355000 (file offset 0x400).
3. It writes decompressed data to rdi = 0x6000 (within .text section).
4. ebx starts at 0 (xor ebx, ebx).
5. Bit reading is MSB-first via add ebx, ebx / adc ebx, ebx / jb.
6. The first dword 0xFFFF6FF6 has 16 leading 1-bits.
7. This produces 16 consecutive LITERALs.
8. The ground truth needs a MATCH after 5 LITERALs.
